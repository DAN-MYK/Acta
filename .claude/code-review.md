### SQL ін'єкції

Всі SQL запити використовують **параметризовані запити** через `sqlx::query_as!` / `sqlx::query!` з `$1`, `$2` тощо. Конкатенація рядків не використовується в SQL.

**⚠️ `src/db/counterparties.rs:55`** — `format!("%{}%", query)` для ILIKE. Це **не SQL ін'єкція** (значення передається як параметр `$1`), але спецсимволи `%` та `_` у введенні користувача не екрануються. Наприклад, пошук `100%` знайде не лише "100%", а й "100ABC". Некритично для MVP.

### Витоки з'єднань з БД

Транзакції закриваються коректно:
- `src/db/acts.rs:109` — `pool.begin()` → `tx.commit()` на рядку 172. Якщо помилка до commit — tx дропається і робить rollback автоматично. **Коректно.**
- Усі інші запити використовують `pool` напряму — з'єднання повертається в пул автоматично після `fetch_all/fetch_optional/execute`. **Коректно.**

## 4. Slint UI патерни

### Бізнес-логіка в .slint

**✅ Коректно.** Файли `.slint` містять лише:

Жодних обчислень, валідацій, або роботи з даними.

### Дані через `in`/`out` properties

**✅ Коректно.** Всі дані передаються через `in property`, колбеки через `callback`.

### Race conditions в async колбеках

**⚠️ Потенційна проблема: stale state після закриття вікна.**

`src/main.rs` рядки 96, 146, 198, 236 — кожен `tokio::spawn` містить `.unwrap()` на `upgrade_in_event_loop`. Якщо вікно закрити під час виконання async операції:
1. `ui.run()` повертає Ok
2. `main()` завершується
3. Запити в `tokio::spawn` ще виконуються
4. `upgrade_in_event_loop` повертає `Err`
5. `.unwrap()` → **panic в tokio task**

**Рішення:** замінити `.unwrap()` на `let _ =` або `.ok()` — ігнорувати помилку якщо вікно вже закрите.

**⚠️ Подвійний запит при advance_status.**

`src/main.rs:180-199` — `advance_status` виконує запит, потім одразу `list` для оновлення UI. Якщо користувач швидко натисне кнопку двічі — два `advance_status` можуть виконатися паралельно, і акт перескочить два статуси. `change_status` перевіряє допустимість переходу, тому другий виклик поверне помилку — **фактичного подвійного просування не буде**, але може бути `tracing::error` в логах. Некритично.

## 6. Якість коду

### Дублювання коду

**⚠️ `src/main.rs` — `build_models` та `build_acts_models`** (рядки 321-345 та 352-372) — майже ідентична логіка конвертації `Vec<Vec<SharedString>>` → `ModelRc<ModelRc<StandardListViewItem>>`. Різниця лише в додатковому полі `statuses`. Можна виділити спільну функцію для конвертації рядків.

**⚠️ Однакові SQL SELECT поля в counterparties.rs** — рядки 18-20, 37-39, 59-60, 82-84, 110-112, 163-164 повторюють один і той же список полів. Це типова проблема sqlx — макрос потребує літеральний SQL. Не можна виділити в константу. Допустимо.

**⚠️ Патерн `upgrade_in_event_loop` + `.unwrap()`** повторюється 4 рази в `main.rs` (рядки 88-96, 139-146, 190-198, 229-236) з однаковою структурою. Можна було б виділити helper, але для 4 випадків це не критично.

### Dead code

**⚠️ `src/models/mod.rs:6-7`** — `#[allow(unused_imports)]` явно вказує що є невикористані імпорти: `Act`, `ActItem`, `NewActItem`, `UpdateAct` реекспортуються, але ніде не імпортуються через `models::Act` тощо. Код `db/acts.rs` використовує прямий шлях `crate::models::act::*`.

**⚠️ `src/db/counterparties.rs:77-100`** — функція `create()` не використовується в `main.rs`. Аналогічно `update()` (рядки 103-137) та `find_by_bas_id()` (рядки 160-175). Це підготовлені CRUD операції — **dead code, але очікувано** для MVP.

**⚠️ `src/db/acts.rs:178-204`** — `update()` не викликається з `main.rs`. Те саме для `get_by_id()` (рядки 65-99) та `change_status()` (рядки 210-248 — використовується тільки через `advance_status`). Очікуваний dead code.

**⚠️ `src/bin/migrate.rs`** — заглушка з `// TODO: логіка імпорту BAS файлів`. Весь файл фактично нічого не робить окрім парсингу аргументів.

### Коментарі та навчальність

**✅ Відмінна робота.** Код рясно задокументований українською:
- Пояснення Rust концепцій: `Rc` vs `Arc`, `Send`, `upgrade_in_event_loop`, `&mut *tx`, `let-else`
- Пояснення sqlx специфіки: `query_as!`, `RETURNING`, `rows_affected()`
- Пояснення Slint патернів: `StandardListViewItem`, `ModelRc`, properties vs callbacks
- Коментарі до SQL: `ILIKE`, `NOW()`, `CASCADE`
